## Что проверять (общее)

1. Найди все изменённые файлы через `git diff`
2. Проверь:
   - соответствие naming conventions проекта
   - наличие тестов для новой логики
   - отсутствие хардкода конфигурации
   - новые зависимости добавлены в `requirements.txt`
3. Собери замечания в структурированный список
4. Если замечаний нет, напиши короткое подтверждение

## Специфика проекта (FastAPI гидродинамический симулятор)

- `templates.TemplateResponse(request, name, context)` — Starlette 1.1.0 требует `request` **первым** аргументом. Старый код: `TemplateResponse(name, {"request": request, ...})`. Ошибка при неверном порядке: `TypeError: unhashable type: 'dict'`
- `MODEL_STATE` — глобальный dict, **состояние теряется при перезапуске сервера**. Не полагаться на него между рестартами
- `file_processing()` — новый код должен корректно парсить `*` repeat-count notation (например `3*0.25`). Если в файле появился новый парсер, убедись, что он тоже это поддерживает
- Плагины сохраняются в `static/plots/<model_name>/` — путь должен существовать (`os.makedirs`). Возвращается URL вида `/static/plots/...`
- `compute_cell_geometry()` — использует ConvexHull (`scipy.spatial`), чувствителен к размерности COORD (6 чисел на узел) и ZCORN (8 углов на ячейку). Ошибки обрабатываются через `warnings.warn`
- `xtgeo` импортирован, но не используется в коде — не обязательно устанавливать для работы
- Нет тестовой инфраструктуры — проверяй изменения вручную через браузер (http://localhost:8000)
- Код на русском (комментарии, UI) и английском (переменные, функции) — не смешивать стили в рамках одного файла

## Чего не делать

- Не предлагай стилистические правки, которые покрывает линтер
- Не переписывай код, только укажи на проблему
- Не предлагай добавлять тестовую инфраструктуру — её нет в проекте
